
# راهنمای خلاصه و جامع گیت (بوت‌کمپ ۱۴۵-هوش مصنوعی)

در این مستند یک راهنمای کاربردی و به‌روز برای کاروندان بوت‌کمپ ۱۴۵ مکتب شریف آماده شده است تا با دستورات مهم و پرکاربرد گیت آشنا شوند.

## 🛠️ تنظیمات اولیه و پیکربندی (Setup & Configuration)
تنظیم اطلاعات کاربری، مقادیر پیش‌فرض و تنظیمات دلخواه رابط کاربری در تمام مخازن محلی.

```bash
# تنظیم نام متصل به تراکنش‌های کامیت شما در مخزن فعلی
git config user.name "[name]"

# تنظیم ایمیل متصل به تراکنش‌های کامیت شما در مخزن فعلی
git config user.email "[email]"

# تنظیم نام متصل به تراکنش‌های کامیت شما در تمام مخازن 
git config --global user.name "[name]"

# تنظیم ایمیل متصل به تراکنش‌های کامیت شما در تمام مخازن 
git config --global user.email "[email]"

# پیکربندی گیت برای استفاده از 'main' به عنوان نام شاخه پیش‌فرض در تمام مخازن جدید
git config --global init.defaultBranch main

# فعال‌سازی رنگ‌بندی خودکار خروجی خط فرمان برای بررسی راحت‌تر چشمی
git config --global color.ui auto
```

## 📁 ایجاد مخازن (Creating Repositories)
راه‌اندازی یک مخزن کاملاً جدید یا کلون کردن یک مخزن موجود از یک سرور مثلا گیت‌هاب یا سرور شرکتی که در آن کار می‌کنید.

```bash
# راه‌اندازی یک پوشه موجود به عنوان مخزن گیت
git init

# دانلود کامل یک مخزن از یک آدرس راه دور (شامل تمام فایل‌ها، تاریخچه و شاخه‌ها)
git clone [url]
```

## 📝 اعمال تغییرات و ثبت وضعیت (Making Changes & Snapshotting)
کار با محیط آماده‌سازی (Staging Area)

برای بررسی تفاوت‌ها و ردیابی تغییرات.

```bash
# مشاهده وضعیت خلاصه و فشرده از فایل‌های تغییریافته و آماده‌سازی‌شده
git status -sb

# آماده‌سازی (Stage) یک فایل خاص برای قرارگیری در کامیت بعدی
git add [filename]

# تمام فایل‌های جدید و تغییر یافته در مسیر جاری را stage می‌کند.
# مواردی که صریحا در فایل (.gitignore) ذکر شده باشند در حالت آماده‌سازی (Stage) قرار نمی‌گیرند.
git add .

# تمام تغییرات (افزودن، حذف و ویرایش) را در کل مخزن stage می‌کند.
git add -A

# نمایش تفاوت‌های محتوایی برای فایل‌هایی که تغییر کرده‌اند اما هنوز آماده‌سازی نشده‌اند
git diff

# نمایش تفاوت‌های محتوایی بین فایل‌های آماده‌سازی‌شده و آخرین کامیت ثبت‌شده
git diff --staged

# ثبت دائمی وضعیت آماده‌سازی‌شده (Staged) در تاریخچه نسخه‌های پروژه
git commit -m "[descriptive  commit message]"
```

همانطور که در کلاس گفته شد معمولا تیم‌های توسعه نرم‌افزار برای پیامهای کامیت قواعدی تعریف می‌کنند تا معنی‌دار و قابل فهم باشند. بعنوان مثال به موارد ذیل توجه کنید:

## 🙈 نادیده گرفتن فایل‌ها (Ignoring Files with .gitignore)
مشخص کردن فایل‌هایی که گیت باید به طور عمدی آن‌ها را نادیده بگیرد و ردیابی نکند.

فایل `.gitignore` یک فایل متنی ساده (بدون نام و فقط با پسوند `.` در ابتدا) است که در پوشه اصلی  پروژه شما قرار می‌گیرد. این فایل به گیت دستور می‌دهد که برخی از فایل‌ها یا پوشه‌ها را کاملاً نادیده بگیرد، طوری که انگار اصلاً در پروژه وجود ندارند.


### چرا به `.gitignore` نیاز داریم؟
ما نمی‌خواهیم یا نباید همه فایل‌های موجود در پوشه پروژه را روی گیت‌هاب یا مخزن پروژه بفرستیم. این فایل‌ها معمولاً به چند دسته تقسیم می‌شوند:
1. **اطلاعات حساس (Credentials):** مثل فایل‌های `.env` که حاوی رمز عبور دیتابیس یا کلیدهای API (API Keys) هستند. لو رفتن این فایل‌ها خطر امنیتی شدیدی دارد.
2. **فایل‌های سیستمی (OS Files):** مثل فایل `.DS_Store` در مکینتاش یا `Thumbs.db` در ویندوز که توسط سیستم‌عامل ساخته می‌شوند و ربطی به پروژه ندارند.
3. **خروجی‌های کامپایل (Build Outputs):** مثل پوشه‌های `dist` یا `build` یا فایل‌های اجرایی مثل `.exe` و فایل‌های لاگ سرور (`.log`).

### نحوه نگارش الگوها (Syntax Patterns) در `.gitignore`
- **نادیده گرفتن یک فایل خاص:**
```bash
tokens.txt
```
- **نادیده گرفتن تمام فایل‌ها با یک پسوند خاص: (مثلاً تمام فایل‌های لاگ)**
```bash
*.log
```
-**نادیده گرفتن کل یک پوشه و محتویات آن:** (حتماً علامت / را در انتها بگذارید)
```bash
data/
```
-**ایجاد استثنا (نفی کردن):** اگر می‌خواهید تمام فایل‌های لاگ نادیده گرفته شوند به جز یک فایل خاص، از علامت ! استفاده کنید:
```bash
*.log
!important.log
```
-**فقط در پوشه ریشه:** اگر بنویسید ``/todo.txt`` فقط فایل خط اول پروژه نادیده گرفته می‌شود، اما اگر در زیرپوشه‌ها فایلی به نام ``todo.txt`` باشد، گیت آن را ردیابی می‌کند.


**⚠️ نکته بسیار حیاتی**: "فایل من در ```.gitignore``` است اما گیت هنوز آن را ردیابی می‌کند! چرا؟!"
- این یکی از رایج‌ترین مشکلات برنامه‌نویسان است. قانون گیت این است: اگر فایلی از قبل توسط گیت ردیابی شده باشد (یعنی حداقل یک بار کامیت شده باشد)، اضافه کردن نام آن به فایل ‍‍‍```.gitignore``` هیچ تأثیری روی آن نخواهد داشت! گیت همچنان تغییرات آن را زیر نظر می‌گیرد.
- **راه حل:** باید به گیت بگویید این فایل را از حافظه (Cache) خود پاک کند، بدون اینکه فایل واقعی از روی کامپیوتر شما حذف شود. برای این کار مراحل زیر را بروید:
۱. ابتدا نام فایل را در ```.gitignore``` بنویسید و ذخیره کنید.
۲. دستور زیر را در ترمینال بزنید تا فایل از کش گیت حذف شود:
```bash
git rm --cached .env
```
(به جای ```.env``` نام فایل خود را بگذارید. اگر یک پوشه کامل است از دستور ```git rm -r --cached folder_name/``` استفاده کنید).

۳. حالا این تغییر را کامیت کنید:
```bash
git add .
git commit -m "removing the file from git tracking"
```

از این لحظه به بعد، فایل شما به طور امن روی کامپیوترتان باقی می‌ماند اما گیت دیگر کاری به کار آن ندارد.

## 💡 لغو تغییرات و خارج کردن از حالت stage

در نسخه‌های قدیمی گیت، از دستورات git reset و git checkout برای حذف تغییرات یا خارج کردن فایل‌ها از حالت آماده‌سازی استفاده می‌شد که به دلیل چندمنظوره بودن، پتانسیل حذف تصادفی داده‌ها را داشت.
گیت مدرن از دستور صریح و ایمن git restore استفاده می‌کند.

```bash
# خارج کردن فایل از حالت آماده‌سازی
# تغییرات کد محلی را حفظ می‌کند اما آن‌ها را از محیط Staging خارج می‌سازد
git restore --staged [filename]

# حذف تغییرات محلی
# تغییرات کامیت‌نشده یک فایل در پوشه کاری را کاملاً پاک کرده و آن را به وضعیت آخرین کامیت برمی‌گرداند
git restore [file]
```

## 🌿 مدیریت شاخه‌ها (Managing Branches)
جداسازی مسیرهای کاری، سوییچ ایمن بین تسک‌ها و ادغام تغییرات.

```bash
# ساخت شاخه جدید
git branch branchname
# یا دستور مدرن‌تر
git switch -c branchname

# لیست کردن تمام شاخه‌های محلی در مخزن فعلی
git branch

# لیست کردن شاخه‌های موجود روی سرور (Remote Branches)علاوه بر شاخه‌های محلی
git branch -a



# حذف ایمن شاخه محلی مشخص‌شده (اگر تغییرات ادغام‌نشده داشته باشد، هشدار می‌دهد)
git branch -d [branchname]

# ادغام تاریخچه شاخه مشخص‌شده در شاخه فعال فعلی شما
git merge [branchname]
```

### جابه‌جایی بین شاخه‌ها

<div dir="rtl">

| دستور قدیمی / سنتی | دستور مدرن | نوع عملیات |
|-------------------|------------|------------|
| `git checkout [branch-name]` | `git switch [branch-name]` | سوییچ به یک شاخه موجود |
| `git checkout -b [branch-name]` | `git switch -c [branch-name]` | ایجاد و سوییچ فوری به شاخه جدید |
| `git checkout -` | `git switch -` | بازگشت به شاخه قبلی که در آن بودید |

## 🔍 بررسی تاریخچه و بازبینی (Reviewing History & Inspecting)
مرور، جستجو و پیمایش در روند تکامل پروژه.
```bash
# لیست کردن تاریخچه نسخه‌های استاندارد برای شاخه فعال فعلی
git log

# خلاصه‌سازی گزارش کامیت‌ها در خطوط مجزا و کوتاه برای مرور سریع
git log --oneline

# نمایش گرافیکی تاریخچه شاخه‌ها برای مشاهده ساختار شاخه‌ها، مرج‌ها و مسیر تکامل پروژه
git log --oneline --graph --all

# نمایش تاریخچه تغییرات یک فایل خاص و ردیابی دقیق آن حتی در صورت تغییر نام در گذشته
git log --follow [filename]

# نمایش متادیتا (اطلاعات جانبی) و تفاوت‌های دقیق کد در یک کامیت مشخص
git show [commit-id]
```

## 🚚 ردیابی تغییرات مسیر و فایل‌ها (Tracking Path Changes)
مدیریت فایل‌هایی که جابه‌جا، حذف یا از ردیابی گیت خارج می‌شوند.

```bash
# حذف فایل از دیسک محلی و آماده‌سازی حذف آن برای کامیت بعدی
git rm [نام-فایل]

# حذف یک فایل از فرآیند ردیابی گیت، در حالی که فایل به صورت محلی روی سیستم شما باقی می‌ماند
git rm --cached [نام-فایل]

# تغییر مسیر یا نام یک فایل و آماده‌سازی خودکار این تغییر مسیر
git mv [مسیر-قدیمی] [مسیر-جدید]
```


## 📦 ذخیره‌سازی موقت تغییرات (Stashing)
بایگانی موقت تغییرات نیمه‌کاره برای پاک‌سازی فوری محیط کار و بازگشت به آن در آینده.

```bash
# ذخیره موقت تمام فایل‌های تغییریافته و ردیابی‌شده برای تغییر وظیفه بدون نیاز به کامیت کردن
git stash

# برای اینکه فایلهای جدید هم ذخیره شوند
git stash --include-untracked

# لیست کردن تمام تغییرات ذخیره‌شده‌ی موقت به ترتیب زمانی
git stash list

# اعمال تغییرات از بالاترین عضو صفِ stash و حذف هم‌زمان آن آیتم از لیست
git stash pop

# حذف بالاترین آیتم stash بدون اعمال تغییرات کد آن بر روی پروژه
git stash drop
```

## 🔄 همگام‌سازی تغییرات (Synchronizing Changes)
اشتراک‌گذاری داده‌ها و به‌روزرسانی‌ها بین مخزن محلی شما و سرورهای میزبان (GitHub, GitLab, Company Git Server).

```bash
# متصل کردن آدرس URL یک مخزن راه دور به یک نام مستعار محلی (معمولاً 'origin')
git remote add [alias] [url]

# دانلود تمام تاریخچه و شاخه‌های ردیابی از ریموت بدون ادغام (Merge) آن‌ها در فضای کاری محلی
git fetch [alias]

# دریافت به‌روزرسانی‌ها از ریموت و ادغام فوری آن‌ها در شاخه محلی فعلی شما
git pull [alias] [branchname]

# آپلود کامیت‌های شاخه محلی شما به مخزن راه دور مشخص‌شده
git push [alias] [branchname]
```
### 🔎 مشاهده مخازن راه دور (Viewing Remote Repositories)
گاهی لازم است بررسی کنید مخزن محلی شما به چه آدرس‌هایی متصل است یا مطمئن شوید Remote را به درستی تنظیم کرده‌اید. برای این کار از دستور زیر استفاده می‌شود:
```bash
git remote -v
```
خروجی این دستور نام مستعار (Alias) هر Remote و آدرس آن را برای عملیات دریافت (Fetch) و ارسال (Push) نمایش می‌دهد.
```bash
origin https://github.com/example/project.git (fetch)
origin https://github.com/example/project.git (push)
```



## بازنویسی تاریخچه (Rewriting History)
بخش بازنویسی تاریخچه (Rewriting History) یکی از قدرتمندترین و در عین حال حساس‌ترین بخش‌های گیت است. این ابزارها به شما اجازه می‌دهند قبل از اینکه کدهای خود را با دیگران به اشتراک بگذارید (Push کنید)، ظاهر تاریخچه پروژه را تمیز، مرتب و خوانا کنید.

1. ۱. دستور ```git commit --amend``` (اصلاح آخرین کامیت)

وقتی کامیتی را ثبت می‌کنید و بلافاصله متوجه یک اشتباه می‌شوید، این دستور نجات‌دهنده شماست. کلمه Amend به معنای «اصلاح و تجدید نظر» است. این دستور به جای ساخت یک کامیت جدید، آخرین کامیت شما را باز می‌کند، تغییرات جدید را داخل آن می‌گذارد و آن را دوباره می‌بندد.
```bash
git commit --amend
```
چه زمانی از آن استفاده کنیم؟

1. اشتباه تایپی در پیام کامیت: کامیتی ثبت کرده‌اید اما پیام آن غلط املایی دارد یا ناقص است.

2. فراموش کردن یک فایل یا یک خط کد: کامیت را ثبت کرده‌اید اما یادتان رفته فایلی را اضافه کنید یا یک تغییر کوچک (مثل پاک کردن یک console.log) را اعمال کنید.

نحوه استفاده:

- **فقط تغییر پیام کامیت:**
اگر فقط می‌خواهید متن پیام آخرین کامیت را عوض کنید، کافی است بنویسید:
```bash
git commit --amend -m "پیام جدید و اصلاح شده"
```

- **اضافه کردن تغییرات جدید به کامیت قبلی:**
اگر فایلی جا مانده است، ابتدا تغییرات را اعمال کرده، آن را آماده‌سازی (Stage) کنید و سپس دستور اصلی را بزنید:
```bash
git add file_nam_shode.txt
git commit --amend --no-edit
```

(نکته: عبارت --no-edit یعنی تغییرات جدید را اضافه کن اما پیام کامیت قبلی را دست نزن).

⚠️ **نکته بسیار مهم**	: این دستور شناسه (SHA) کامیت قبلی را کاملاً عوض می‌کند. بنابراین هرگز کامیتی که قبلاً آن را Push کرده‌اید و دیگران آن را دریافت کرده‌اند را amend نکنید، زیرا تاریخچه محلی شما با سرور ناسازگار می‌شود.


۲. دستور ```git rebase [branchname]``` (تغییر پایه یا بازآرایی شاخه)

برای درک rebase، ابتدا باید بدانید کلمه base به معنای «پایه یا ریشه» است. وقتی شما از شاخه اصلی (مثلاً ```main```) یک شاخه جدید برای خودتان می‌سازید (مثلاً ```feature```)، ریشه شاخه شما آخرین کامیت آن لحظه در ```main``` است.

در طول زمانی که شما روی شاخه خود کار می‌کنید، همکارانتان کامیت‌های جدیدی را روی شاخه ```main``` می‌فرستند. حالا شاخه شما از نظر زمانی عقب افتاده است. دستور ```git rebase main``` می‌آید و تمام کامیت‌های اختصاصی شاخه شما را موقتاً کنار می‌گذارد، پایه (ریشه) شاخه شما را به آخرین کامیت فعلی ```main``` متصل می‌کند، و سپس کامیت‌های شما را یکی‌یکی روی آن اعمال می‌کند.

```bash
git switch feature


main:    A---B---C
               \
feature:         D---E

```

```bash

Before rebase:
main:    A---B---C---F---G
               \
feature:         D---E


git rebase main


After rebase:
main:    A---B---C---F---G
                           \
feature:                    D'---E'

```


### ⭐ دریافت تغییرات با Rebase (روش پیشنهادی برای شاخه‌های شخصی)
در حالت عادی، دستور ```git pull``` در واقع معادل اجرای دو دستور زیر است:
```bash
git fetch
git merge
```
یعنی ابتدا تغییرات جدید را از سرور دریافت می‌کند و سپس آن‌ها را با شاخه محلی شما Merge می‌کند. در نتیجه ممکن است کامیت‌های اضافی از نوع Merge Commit در تاریخچه پروژه ایجاد شوند.

اگر می‌خواهید تاریخچه پروژه تمیزتر و خطی باقی بماند، می‌توانید از گزینه ```--rebase``` استفاده کنید:
```bash
git pull --rebase origin main
```
در این حالت گیت:

1. تغییرات محلی شما را موقتاً کنار می‌گذارد.
2. آخرین تغییرات شاخه مقصد را از سرور دریافت می‌کند.
3. شاخه محلی شما را روی آخرین نسخه شاخه مقصد قرار می‌دهد.
4. کامیت‌های محلی شما را مجدداً روی آن اعمال می‌کند.

به این ترتیب به جای ایجاد Merge Commit اضافی، تاریخچه پروژه به صورت خطی و خواناتر باقی می‌ماند.

### حالا ببینیم مرج چه تفاوتی با ریبیس دارد؟

این مثال فرض می‌کند merge روی شاخه feature انجام شده است.

```bash

git switch feature
git merge main

Before merge:
main:    A---B---C---F---G
               \
feature:         D---E


git merge main


After merge:
main:    A---B---C---F---G
               \         \
feature:         D---E----M
```





## چرا به جای ```merge``` از ```rebase``` استفاده کنیم؟

زمانی که شما دو شاخه را با ```git merge``` ادغام می‌کنید، گیت یک کامیت اضافی به نام Merge Commit برای اتصال آن‌ها می‌سازد که باعث می‌شود درخت تاریخچه گیت (گراف) شلوغ و مارپیچ شود. اما ```rebase``` تاریخچه را طوری بازنویسی می‌کند که انگار شما از همان اول، کارتان را روی آخرین نسخه شاخه اصلی شروع کرده بودید. نتیجه این کار یک تاریخچه کاملاً خطی، صاف و تمیز است.

### قانون طلایی ریبیس (The Golden Rule of Rebasing)
«هیچ‌وقت روی شاخه‌های عمومی و اشتراکی، ریبیس نکنید!»

اگر روی شاخه‌ای کار می‌کنید که کدهای آن روی گیت‌هاب قرار دارد و افراد دیگر هم دارند روی همان شاخه کد می‌زنند، دستور rebase تاریخچه را برای همه آن‌ها خراب می‌کند و منجر به تداخل‌های (Conflict) بسیار پیچیده می‌شود. 
روی branchهایی که دیگران روی آن کار می‌کنند rebase نکنید.


##  هوک‌های گیت (Git Hooks)
خودکارسازی اجرای اسکریپت‌های سفارشی قبل یا بعد از رویدادهای مهم گیت (مانند کامیت، پوش یا مرج).

هوک‌ها (Hooks) در لغت به معنای «قلاب» هستند. در دنیای نرم‌افزار، هوک‌ها اسکریپت‌هایی (کدهایی) هستند که گیت به طور خودکار وقتی اتفاق خاصی می‌افتد، آن‌ها را اجرا می‌کند. این ویژگی به شما اجازه می‌دهد تا استانداردهای خاصی را به صورت اجباری در پروژه پیاده کنید.

### هوک‌ها کجا قرار دارند؟
وقتی یک مخزن گیت می‌سازید، یک پوشه مخفی به نام `.git` ایجاد می‌شود. اگر داخل آن بروید، پوشه‌ای به نام `hooks` را می‌بینید (`.git/hooks/`). 
داخل این پوشه، گیت چند فایل نمونه با پسوند `.sample` قرار داده است (مثل `pre-commit.sample`). برای فعال کردن هرکدام، کافی است پسوند `.sample` را حذف کنید و کدهای خود را (به زبان Bash، Python، Node.js یا هر زبان دیگری) داخل آن بنویسید.

```bash
# حیاتی: نادیده گرفتن و دور زدن هوک‌های pre-commit و commit-msg در حین کامیت (با احتیاط استفاده شود!)
git commit -m "[message]" --no-verify

# تغییر پوشه محلی که گیت در آن به دنبال هوک‌ها می‌گردد (عالی برای اشتراک‌گذاری هوک‌ها در پروژه)
git config core.hooksPath .githooks
```

### کاربردی‌ترین هوک‌های گیت کدامند؟

هوک‌ها به دو دسته **سمت کاربر (Client-side)** و **سمت سرور (Server-side)** تقسیم می‌شوند. محبوب‌ترین هوک‌های سمت کاربر عبارتند از:

۱. **`pre-commit` (قبل از کامیت):**
   این هوک دقیقاً بعد از زدن دستور `git commit` و *قبل* از اینکه پیام کامیت را بنویسید اجرا می‌شود. 
   * **کاربرد:** بررسی فرمت کدهای شما (Linter یا Prettier)، اجرای تست‌های واحد (Unit Tests) پروژه. اگر تست‌ها خطا بدهند، گیت فرآیند کامیت را متوقف می‌کند و اجازه ثبت کامیت را نمی‌دهد تا کدهای خراب کامیت نشوند.

۲. **`commit-msg` (بررسی پیام کامیت):**
   این هوک پس از نوشتن پیام کامیت اجرا می‌شود و متن پیام را بررسی می‌کند.
   * **کاربرد:** اجبار کردن تیم به استفاده از استانداردهای نوشتن پیام کامیت (مثل استاندارد Conventional Commits). برای مثال اگر پیام کامیت شامل ساختار مشخصی نباشد، کامیت رد می‌شود.

۳. **`pre-push` (قبل از ارسال به سرور):**
   دقیقاً قبل از اینکه کدها به گیت‌هاب یا سرور منتقل شوند اجرا می‌شود.
   * **کاربرد:** جلوگیری از Push مستقیم روی شاخه `main` یا اجرای تست‌های سنگین‌تر (Integration Tests) برای اطمینان از اینکه کدهای ارسالی کاملاً سالم هستند.
   
### چالش بزرگ هوک‌ها و راه حل مدرن آن

یک چالش بزرگ در گیت وجود دارد: **پوشه `.git` هرگز کامیت نمی‌شود و روی گیت‌هاب نمی‌رود.** بنابراین، اگر شما یک هوک عالی برای پروژه بنویسید، همکاران شما آن را دریافت نخواهند کرد!

**راه‌حل چیست؟**
در نسخه‌های مدرن گیت، شما می‌توانید پوشه هوک‌ها را تغییر دهید. 
۱. یک پوشه معمولی در پروژه خود بسازید (مثلاً به نام `.githooks`).
۲. هوک‌های خود را داخل آن بنویسید و آن را کامیت کنید تا همه همکاران کدهای هوک را دریافت کنند.
۳. حالا هر فرد در سیستم خود دستور زیر را اجرا می‌کند تا گیت به جای پوشه مخفی، از این پوشه جدید استفاده کند:
```bash
git config core.hooksPath .githooks
```

**⚠️ ترفند فرار: دستور --no-verify چیست؟**

گاهی اوقات کدی زده‌اید که عجله دارید آن را کامیت کنید، اما هوک ```pre-commit``` به خاطر یک خطای کوچک در ظاهرِ کد (Formatting) یا تستی که در حال حاضر مهم نیست، جلوی کامیت شما را می‌گیرد. برای دور زدن و غیرفعال کردن موقتی هوک‌ها در آن کامیت خاص، کافیست پرچم ```--no-verify``` را در انتهای دستور کامیت اضافه کنید:
```bash
git commit -m "کامیت فوری بدون اجرای هوک" --no-verify
```
